# Batch Schnorr Baseline Status

Status: reviewed public BIP-340 batch verification baseline unavailable in this
package.

This document records the batch-Schnorr evidence used by the pre-review
benchmark package. It is an evidence/status note, not a consensus
implementation plan.

## Reviewed Public API Status

The bundled upstream libsecp256k1 commit
`aea86bc3509f3ceb5c047fe6567bf8910f72a728` exposes public BIP-340 individual
verification through `secp256k1_schnorrsig_verify`, but it does not expose a
reviewed public BIP-340 batch verification API.

The bundled upstream test material still treats `verify_batch` as TODO. The
release evidence generated by `scripts/check_batch_schnorr_baseline.py` records
this as:

- reviewed public batch API: unavailable;
- reviewed public batch timing baseline: unavailable;
- experimental batch baseline: separate sensitivity analysis only.

The generated report therefore keeps:

- `.benchmarks.schnorr_bip340.batch_reviewed_public_api.status =
  "unavailable"`;
- `.benchmarks.schnorr_bip340.batch.status = "unavailable"` as the
  reviewed-public alias.

## Upstream Branch Evidence

When network checking is enabled, `scripts/check_batch_schnorr_baseline.py`
queries upstream `bitcoin-core/secp256k1` heads matching batch/Schnorr/BIP-340
review candidates. If the query times out or cannot complete, the script reports
`evidence-incomplete` rather than silently treating the baseline as unavailable.

Ordinary local release checks run with `--skip-upstream` to avoid making
publication readiness depend on network access. Network-checked evidence can be
regenerated with:

```sh
QRS_RELEASE_NETWORK_BATCH=1 bash scripts/release_check.sh
```

## Experimental Baseline

The benchmark package also includes an experimental native BIP-340 batch
verifier for sensitivity analysis. It uses libsecp256k1 internals, deterministic
nonzero coefficients derived from a domain-separated hash of all batch inputs,
and `a_0 = 1`.

This implementation is not a reviewed public libsecp256k1 API and must not be
treated as consensus code or as reviewed activation evidence. It is reported
separately under `.benchmarks.schnorr_bip340.batch_experimental*`.

Experimental batch timing includes:

- signature and public-key parsing;
- BIP-340 challenge computation;
- coefficient derivation;
- multiscalar batch equation checking;
- invalid-batch rejection without timing fallback to individual verification.

## Resource-Accounting Use

The resource-accounting decision keeps reviewed-public batch Schnorr distinct
from the experimental baseline. A slow experimental batch baseline is not
favorable evidence for QRS because it does not tighten the individual Schnorr
baseline.

The decision output therefore reports hypothetical reviewed-batch speedups
separately. It must never synthesize a fake batch baseline by dividing individual
Schnorr timings by an assumed speedup.

## Regeneration

```sh
python3 scripts/check_batch_schnorr_baseline.py \
  --json out/batch-evidence.json \
  --markdown out/batch-evidence.md \
  --upstream-timeout-seconds 15
```

For normal offline release checks, omit upstream network evidence by relying on
`bash scripts/release_check.sh`, which generates temporary batch evidence with
`--skip-upstream`.
