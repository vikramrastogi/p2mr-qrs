# QRS Native Benchmark Report Template

This template is generated by `qrs_native_bench --markdown`, but reviewers can
use it to check that a published report contains the required evidence.

## Conclusion

Use one of the conclusion strings from `docs/internal/implementation-notes/report-generation.md`; do not infer
consensus readiness from a single machine.

## Environment

- CPU model
- CPU flags
- OS
- Compiler and flags
- Frequency governor / power mode
- OpenSSL version or commit
- libsecp256k1 version or commit
- Benchmark commit

## Raw Timing Table

Include valid QRS, invalid fixed-length QRS, individual Schnorr, invalid
Schnorr, experimental batch Schnorr where available, and reviewed-public batch
Schnorr status. Report mean, stddev, median, p95, p99, maximum, and throughput.
Timing statistics are computed over per-batch per-operation means, not
individual operation latency samples.

## Block Model

Show the simplified per-input saturated-block model and clearly label it as a
model, not a Bitcoin Core validation trace.

## Caveats

State unavailable baselines, provider/FIPS status, frequency-scaling behavior,
and any experimental batch implementation or branch used. Experimental batch
Schnorr is sensitivity analysis only unless it comes from a reviewed public
libsecp256k1 API.

## Batch Baseline Evidence

Attach `out/batch-evidence.json` and `out/batch-evidence.md` from
`scripts/check_batch_schnorr_baseline.py`. The report is not batch-proof-quality
unless that evidence status is either `unavailable` with a successful upstream
head search or `available` from a reviewed implementation. If the upstream
lookup times out, the evidence status must be `evidence-incomplete`.

## Experimental Batch Sensitivity

When present, report `batch_experimental_valid`,
`batch_experimental_invalid`, and
`schnorr_experimental_batch_saturated_block` separately from reviewed-public
batch status. The report must say that the experimental native BIP-340 batch
baseline is not a reviewed public libsecp256k1 API. It must also state whether
timing includes parsing, challenge computation, coefficient derivation, and the
multiscalar equation check.
