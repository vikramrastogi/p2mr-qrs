# P2MR QRS BIP Pre-Review Package

Status: Draft-stage pre-review package for v0.9.0. This is not
activation-ready consensus evidence.

Repo / branch:
- Repo: `p2mr-qrs`.
- Branch: `main`.

Exact goal:
- Publish a narrow pre-review package for the P2MR Quantum-Rescue Leaf BIP.
- Keep the proposal scoped to one P2MR-only future leaf, one SLH-DSA-SHA2-128s signature, no legacy-output policy, no witness discount, no algorithm registry, and no tapscript opcode in v1.
- Produce evidence for whether QRS validation can safely rely on existing SegWit weight accounting or needs an explicit per-QRS consensus validation budget.

Files to inspect first:
- `bip-p2mr-quantum-rescue-leaf-consolidated.mediawiki`
- `docs/bip-p2mr-quantum-rescue-leaf-v0.9.0.mediawiki`
- `bench_qrs_verify_preliminary.py` or `1-qrs-verify-benchmark.py`
- Existing build files, if any
- Local copies/submodules of OpenSSL and libsecp256k1, if present

Required behavior:
- Create a native benchmark harness in C or C++.
- Benchmark SLH-DSA-SHA2-128s verification using a native FIPS 205 implementation.
- Benchmark BIP-340 Schnorr verification using libsecp256k1.
- Include individual Schnorr verification, an experimental native BIP-340 batch baseline, and a separate reviewed-public batch-Schnorr status field.
- Include valid QRS signatures.
- Include invalid fixed-length QRS signatures that pass structural checks and reach the SLH-DSA verifier.
- Include a Bitcoin-Core-shaped validation loop that models saturated blocks by weight.
- Report median, p95, p99, maximum, mean, standard deviation, and ops/sec.
- Report hardware, CPU model, CPU flags, OS, compiler, compiler flags, OpenSSL version/commit, libsecp256k1 version/commit, and benchmark commit.
- Emit machine-readable JSON and a Markdown report.

Non-goals:
- Do not modify consensus rules.
- Do not change the QRS BIP design.
- Do not benchmark Python wrappers as proof.
- Do not use PQClean SPHINCS+ as the proof backend.
- Do not claim consensus readiness from one machine.
- Do not claim current Bitcoin Core does or does not batch Schnorr unless verified from current source.
- Do not treat the experimental batch verifier as reviewed consensus code.

Implementation constraints:
- Use native code only for proof benchmarks.
- Pin library versions or commits.
- Use release builds with explicit compiler flags.
- Disable frequency-scaling surprises where possible, or report governor / CPU frequency behavior.
- Prevent compiler dead-code elimination by consuming verification results.
- Pre-generate keys, messages, and signatures outside timed loops.
- Include correctness assertions before timing.
- Include invalid signatures generated by flipping bits in otherwise fixed-length signatures.
- Separate structural reject timing from cryptographic invalid-signature timing.
- Keep all benchmark assumptions visible in the JSON report.

Current implementation status:
- OpenSSL SLH-DSA-SHA2-128s backend is implemented through EVP and reports unavailable if the local OpenSSL build does not expose the algorithm or signing API.
- Bundled libsecp256k1 individual Schnorr is wired at commit `aea86bc3509f3ceb5c047fe6567bf8910f72a728`, with `extrakeys` and `schnorrsig` enabled. The harness reports valid and invalid fixed-length individual BIP-340 verification timings.
- Experimental native BIP-340 batch Schnorr is implemented inside the benchmark package using libsecp256k1 internals and the BIP-340 batch equation. It reports valid and invalid fixed-length batch timings for sensitivity analysis only.
- Reviewed public batch Schnorr remains explicit `unavailable` until upstream exposes a reviewed public API. The JSON report never silently omits it, and `scripts/check_batch_schnorr_baseline.py` records the local API/TODO/upstream-head evidence.
- The current BIP draft is included at `docs/bip-p2mr-quantum-rescue-leaf-consolidated.mediawiki`.
- The hardened v0.9.0 BIP draft is included at `docs/bip-p2mr-quantum-rescue-leaf-v0.9.0.mediawiki`.
- The harness writes JSON and Markdown reports and includes the requested saturated-block model fields.
- Provisional structured QRS fixture vectors live in `test_vectors/`; malformed witness and negative cases live in `fuzz/` and `tests/`.
- `docs/REVIEWER_DOSSIER.md` maps predictable objections to evidence, caveats, blockers, or non-goals.
- `docs/RESOURCE_ACCOUNTING_DECISION.md` maps benchmark evidence to the current Draft-stage resource-accounting conclusion.
- `scripts/evaluate_resource_accounting.py` emits a machine-readable pass/fail resource-accounting decision from generated reports.
- `docs/EXPLICIT_QRS_BUDGET_FALLBACK.md` specifies the inactive per-QRS virtual validation-weight fallback if weight-only accounting fails.
- `docs/BITCOIN_CORE_INTEGRATION_REQUIREMENTS.md` defines the Core validation-path artifact required before activation.
- `docs/FINAL_TRANSACTION_VECTOR_SCHEMA.md` and `test_vectors/qrs_transaction_vector.schema.json` define the final serialized-vector shape once BIP-360/QRS definitions are final.
- `docs/REPRODUCIBILITY.md` and `.github/ISSUE_TEMPLATE/benchmark-reproduction.yml` define the independent rerun protocol.
- `docs/RELEASE_CHECKLIST.md` lists the posting checks enforced by `scripts/release_check.sh`.
- `THIRD_PARTY_NOTICES.md` identifies vendored third-party code and license locations.
- `out/sample-native-report.md` is the stable sample Markdown report alias for public review.
- `.github/workflows/native-bench-smoke.yml` builds and runs the quick smoke path on Linux and macOS and uploads the generated reports as CI artifacts.

Tests:
- Build succeeds from clean checkout.
- Valid SLH-DSA signatures verify before timing.
- Invalid fixed-length SLH-DSA signatures fail before timing.
- Valid Schnorr signatures verify before timing.
- Invalid Schnorr signatures fail before timing.
- Invalid fixed-length QRS signatures reach the modeled verifier path and fail cryptographically.
- Structural invalid QRS witnesses fail before the modeled verifier path.
- Benchmark emits JSON.
- Report generator consumes JSON and emits Markdown.
- Re-running the same command overwrites result files deterministically.

Definition of done:
- A reviewer can clone the repo, run one documented command, and obtain a JSON + Markdown report.
- The report directly answers whether QRS validation is slower or faster than individual Schnorr and the experimental batch Schnorr sensitivity baseline under the stated machine and library versions.
- The BIP can cite the artifact as native benchmark evidence without relying on Python-wrapper results.

Current proof status:
- QRS vs individual Schnorr can now be measured.
- QRS vs experimental batch Schnorr can now be measured as sensitivity analysis, not consensus proof.
- QRS vs reviewed public batch Schnorr remains unavailable because the bundled upstream libsecp256k1 commit exposes no reviewed public BIP-340 batch verification API.
- If `scripts/check_batch_schnorr_baseline.py` cannot complete its upstream `git ls-remote` check before `--upstream-timeout-seconds` (default 15), it writes the evidence report and fails closed with `evidence-incomplete` instead of treating network failure as proof.
